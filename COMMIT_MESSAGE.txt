fix: resolve critical search, download, and SAR/optical processing bugs; add multi-sensor grid alignment and GRD extraction support

This session found and fixed a series of real, confirmed bugs across
search filtering, download resilience, and the SAR/optical processing
chain, discovered through live debugging against real USGS/Copernicus
API responses and real Sentinel-1/Landsat data — not synthetic test
cases. Several fixes required correcting an earlier, incomplete fix
once new evidence surfaced. Also adds new library capabilities that
close real gaps found while building the Obuasi, Accra, and Atewa
project notebooks.

## Search & filtering

- providers/usgs.py:
  - Fix spatial filter: query.geometry (polygon AOI) was never checked,
    only query.bbox — searches using geometry= had literally zero
    spatial constraint applied
  - Fix spatialFilter filterType casing: must be lowercase "geojson",
    not "geoJson" — confirmed against a real working M2M client
  - Fix temporal filter: was sent as "temporalFilter", a field name
    the real API doesn't recognize at all; correct field is
    "acquisitionFilter"
  - Fix critical payload structure: acquisitionFilter, spatialFilter,
    and cloudCoverFilter must all nest inside a single "sceneFilter"
    wrapper, not sent as top-level payload keys — this was the root
    cause of searches silently returning USGS's unfiltered
    most-recent-first results regardless of AOI, date range, or cloud
    threshold
  - Fix datetime parsing: was checking non-existent flat fields
    (acquisitionDate/startingDate); real API returns a nested
    temporalCoverage.startDate/endDate structure. Now tries multiple
    formats and logs a warning on genuine parse failure instead of
    silently leaving datetime=None
  - Add defensive client-side date-range filtering as a safety net,
    with a loud warning if the server ever returns out-of-range results
    again

- providers/copernicus.py:
  - Fix spatial filter: same class of bug as USGS — query.geometry was
    never checked, only query.bbox, meaning geometry-based searches
    had zero spatial constraint and could return scenes from anywhere
    on Earth
  - Fix download retry: every failed attempt deleted the partial file
    and restarted from zero; now resumes via HTTP Range requests,
    falling back to a clean restart only if the server doesn't honor
    Range (detected via status code, not assumed)
  - Fix retry backoff: was a flat min(2**attempt, 30) regardless of
    file size; now scales with how much has already downloaded (up to
    120s for multi-GB files), giving a dropped connection meaningful
    recovery time instead of immediately re-hammering it

- providers/aws_earth.py:
  - Fix silent empty-response handling: a 200 OK with an empty body was
    written to disk as a useless 0-byte file and only caught later by
    generic downstream validation; now detected immediately with a
    specific, actionable error message, and the empty file is cleaned
    up rather than left on disk
  - Surface the real per-asset failure reason in DownloadResult.error
    instead of a generic "No assets were successfully downloaded"

- models/satellite_data.py:
  - Fix band alias resolution: "B08" could non-deterministically
    resolve to the B8A asset due to a mislabeled alias table entry
    combined with Python's per-process set iteration order — confirmed
    via testing across multiple PYTHONHASHSEED values

- cli/search_commands.py:
  - Add --product-type and --polarisation as real CLI flags (were
    SearchQuery fields with no CLI exposure at all)
  - Fix --geometry-file: was populating an unused field
    (geometry_geojson) instead of the field every provider actually
    reads (geometry) — polygon AOI searches via CLI were silently
    falling back to a bounding-box approximation

## Download resilience

- core/downloader.py:
  - Fix reproject step: didn't clean up corrupt output on failure, and
    one file failing in a batch silently reverted all successfully
    reprojected files back to their originals
  - Wire all 15 missing spectral indices into --post-process (only
    ndvi was previously connected; others silently no-op'd)
  - Fix file validation: only sampled the first tile of a raster,
    which is exactly the part most likely to survive a truncated
    download — now samples first/middle/last tiles
  - Fix retry logic: a validation failure returned FAILED immediately
    without ever using configured retry_attempts; now retries exactly
    like a raised exception would
  - Fix nodata propagation in ndvi/index computation: cloud-masked or
    fill pixels were computed as real data instead of propagating as
    nodata through to the final index output
  - Remove two invalid `# noqa` directives that didn't reference real
    ruff codes

## Optical + SAR processing

- processing/preprocessor.py:
  - Fix clip(): infinite recursion in _safe_read_1 (self-call typo)
    made cloud_mask/atmos/topo_correct/pansharpen completely
    non-functional
  - Fix clip(): no CRS reprojection at all — a WGS84 AOI clipped
    against a UTM-projected raster (the normal case for real satellite
    imagery) silently computed a nonsensical near-empty window instead
    of raising a clear error
  - Add buffer-and-retry for genuine floating-point precision edge
    cases at raster boundaries, with diagnostic bounds surfaced in the
    error if it still fails
  - Fix clip() nodata handling: cropped-out pixels were never declared
    as nodata in the output profile, so downstream calibration treated
    fill pixels as real data, producing unrealistic extreme values that
    dominated visualization colour scales
  - Add resample(reference=...): align one raster to another's exact
    grid (shape, transform, CRS) — not just matching resolution, which
    can still leave different origins/extents unaligned. Closes a real
    gap where combining results from two different sensors required
    hand-written rasterio.warp code

- processing/sar.py:
  - Add flood_map(detect_direction="decrease"|"increase"|"both"):
    previously only detected the open-water backscatter-decrease
    signature; urban flooding (double-bounce backscatter increase) was
    undetectable by design, not just poorly detected
  - Fix calibrate(): DN=0 fill/nodata pixels were computed into
    physically meaningless -100dB values instead of being excluded

- sar/_native.py:
  - Fix kwargs forwarding: despeckle/calibrate/flood_map/coherence all
    accepted **kw but silently dropped it before calling the real
    implementation — new parameters like detect_direction were
    captured but never actually reached the underlying logic

- sar/extraction.py (new):
  - Add GRDExtractor + georeference_via_gcps_if_needed(): raw Sentinel-1
    measurement TIFFs carry no standard CRS at all — real georeferencing
    is delivered as embedded GCPs. Without this, clip() could "succeed"
    against a meaningless few pixels near the raster origin instead of
    the real AOI, with flood_map() then confidently reporting 0%
    change on data that was never really the target location

- processor/timeseries.py:
  - Add automatic grid alignment to build_index_stack(): different
    acquisitions of the same AOI can come from different satellite
    scene footprints (especially for elongated/irregular AOIs) and
    produce different pixel grids after clipping. Previously raised
    immediately; now reprojects onto the first date's grid by default
    (align_grids=True), with align_grids=False to restore the old
    strict behaviour

## Notebooks / example projects added

- Obuasi mining-belt vegetation trend (real 6-year Landsat time series)
- Accra flood recession mapping (real multi-date Sentinel-1 SAR)
- Atewa Forest Reserve deforestation monitoring template, with combined
  optical (Landsat NDVI) + SAR (Sentinel-1 VH) disturbance detection —
  built as a reusable template for other Forestry Commission reserves
- Accra (GAMA) urban expansion monitoring (NDBI trend)

All notebooks use real, cited administrative/ecological boundaries
(verified against published sources where authoritative shapefiles
weren't programmatically accessible) and contain no synthetic data —
they fail with a clear error rather than substituting placeholder
results when live data is insufficient.
